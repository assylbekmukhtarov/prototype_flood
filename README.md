# Прототип: мониторинг снега и паводков

Работает без GEE и без токенов. Данные Sentinel-2 скачиваются напрямую из AWS через STAC-каталог Element84.

## Лицензии и ограничения на коммерческое использование

| Сервис / данные | Лицензия | Коммерческое использование |
|---|---|---|
| **Open-Meteo** (температура) | CC BY 4.0 | Бесплатно до 10 000 запросов/день. Свыше — нужна коммерческая подписка (~$30/мес). Обязательна атрибуция. |
| **Sentinel-2** (снимки ESA) | Copernicus Data Policy | Свободно для любых целей включая коммерческие. Обязательна атрибуция ESA / Copernicus. |
| **Element84 earth-search** (STAC API) | Публичный, бесплатный | Регистрация не нужна. Нет явных лимитов, но не рекомендуется для высоких нагрузок без договорённости с провайдером. |
| **AWS S3** (COG-файлы Sentinel-2) | AWS Open Data Program | Регистрация не нужна. Данные оплачены ESA в рамках AWS Open Data — чтение бесплатно для всех. Платным становится только исходящий трафик если развернуть собственный сервер на AWS вне региона us-west-2. |
| **OpenStreetMap** (тайлы карты) | ODbL | Данные OSM коммерчески свободны при атрибуции. Публичный тайловый сервер `tile.openstreetmap.org` — только для прототипов и разработки. Для production нужен платный провайдер (Maptiler, Mapbox и др.) или собственный тайловый сервер. |
| **Leaflet** | BSD 2-Clause | Полностью свободна для коммерческого использования. |
| **FastAPI / uvicorn** | MIT | Полностью свободна для коммерческого использования. |
| **rasterio / shapely / pyproj** | BSD / MIT | Полностью свободны для коммерческого использования. |

**Вывод для production:** основные риски — Open-Meteo при высокой нагрузке и OSM тайловый сервер. Остальное коммерчески свободно и не требует регистрации.

## Структура

```
prototype_flood/
├── main.py          # FastAPI: эндпоинты /api/analyze и /api/export-csv
├── sentinel.py      # поиск снимков через STAC, COG-чтение по bbox
├── analysis.py      # NDSI, MNDWI, маски, площади, риск, GeoJSON
├── requirements.txt
└── static/
    └── index.html   # весь фронт: Leaflet карта + форма + результаты
```

## Запуск через Docker

```bash
cd prototype_flood

# собрать образ и запустить
docker compose up --build -d

# или вручную
docker build -t flood-monitor .
docker run -p 8000:8000 flood-monitor
```

Открыть в браузере: http://localhost:8000

Остановить: `docker compose down`

## Установка и запуск

```bash
cd prototype_flood

# создать виртуальное окружение
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

# установить зависимости
pip install -r requirements.txt

# запустить сервер
uvicorn main:app --reload --port 8000
```

Открыть в браузере: http://localhost:8000

## Как пользоваться

1. Нажать на блок **«Нарисуйте прямоугольник»** — на карте активируется режим рисования
2. Нарисовать прямоугольник на нужной территории (2 клика: угол → противоположный угол)
3. Задать даты и пороги в боковой панели
4. Нажать **«Анализировать»** — ждать 5–30 секунд
5. На карте появятся три слоя: снег (синий), вода (голубой), новое затопление (красный)
6. Нажать **«Скачать CSV»** для экспорта метрик


## Ограничения прототипа

- Время ответа: 5–30 сек в зависимости от размера bbox и скорости AWS
- Если за период нет снимков с облачностью < 20% — вернёт ошибку 404
- Историческое среднее снега не реализовано (условие риска №1 пропускается)
- Рекомендуемый размер bbox: не более 1° × 1° (~100 км²)

## Расхождения с Анализом

### 1. Историческое среднее (`historical_years`) не реализовано
ТЗ требует загружать данные за N лет и считать среднее площади снега для той же даты.
Сейчас `historical_avg_km2` всегда `None` — условие риска №1 («снег выше исторического среднего») никогда не проверяется.
Параметр `historical_years` отсутствует в форме и в API.

### 2. Расчёт площади не учитывает overview-downsampling
ТЗ: площадь пикселя = 10 м × 10 м = 0.0001 км². COG читается с уменьшением до 512×512 пикселей (overview), поэтому реальный размер пикселя в памяти уже не 10 м. Площадь считается через `bbox_km2 / total_pixels` — результат приближённый, может отличаться от точного в зависимости от уровня масштабирования.

### 3. Фильтрация шума не реализована
ТЗ (опционально): удалять объекты размером менее 500 пикселей (0.5 га). Не реализовано — на маске могут появляться мелкие шумовые пятна.

### 4. Нет проверки совпадения `date_before` и `date_after`
ТЗ (п. 5.3): если даты совпадают — выводить предупреждение. В `main.py` проверка отсутствует.

### 5. Небольшая погрешность при расчёте нового затопления
При разных размерах масок до/после делается `scipy.ndimage.zoom`, но `real_bbox` снимков может незначительно отличаться. Это вносит погрешность в `compute_new_flood` при сравнении разных сцен.

### 6. RGB-подложка снимка на карте отсутствует
ТЗ (Шаг 7, критерий №9): показывать исходный снимок RGB (B4, B3, B2) как подложку под масками. Сейчас подложка — OpenStreetMap. Реализация требует рендеринга тайлов из COG, что выходит за рамки MVP.
